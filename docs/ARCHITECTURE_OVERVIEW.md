# Architecture Overview

## Current Architecture (Redot Engine LTS — C++)

The existing codebase follows the classic Godot 4.x architecture:

```
+----------------------------------------------------+
|                    Game Logic                       |
|  (GDScript / C# / C++ GDExtension)                 |
+------------------------+---------------------------+
|      Scene Tree        |      Editor               |
|  (Node inheritance)    |  (Docks, Inspector,       |
|  (Reference-counted)   |   Canvas, Debugger)       |
+----+-------------------+----+----------------------+
|    |                     |    |
|    v                     v    v
|  Scene (tree)    +----> Editor (C++)
|  |               |
|  v               |
+--+---------------+----+---------------------------+
|               Servers                              |
|  Rendering | Physics | Audio | Navigation | XR     |
|  (Vulkan)  | (Jolt)  |       |            |       |
+----------------------+-----------------------------+
|               Core / Drivers                       |
|  Variant | String | IO | OS | Thread pool          |
|  Vulkan / GLES3 / Metal / D3D12                    |
+----------------------------------------------------+
|              Platform (Linux, Windows, macOS, ...)  |
+----------------------------------------------------+
```

### Key Characteristics

- **Node tree**: Heavily inheritance-based. Every game object is a `Node` subclass. Properties are stored as typed `Variant` values, often boxed.
- **Memory model**: `RefCounted` reference counting + manual `memnew`/`memdelete`. GC-like patterns via `Ref<>` smart pointers.
- **Scripting**: GDScript (interpreted bytecode) or C# (Mono runtime). Both introduce a GC or interpreter overhead tax.
- **Rendering**: Vulkan 1.x primary, with GLES3/Metal/D3D12 backends. Heavy use of Godot's own material/shader system.
- **Physics**: Jolt Physics (primary, via C++ module) OR Godot Physics 3D (fallback). Jolt is vendored as thirdparty C++.
- **Build system**: SCons (Python). Platform modules are compiled conditionally. No incremental compilation story for engine code.
- **Modules**: 61 optional modules compiled as part of the engine binary. Tight coupling between modules and core.

## Target Architecture (Zodot — Zig)

```
+----------------------------------------------------+
|                    Game / Mods                      |
|  WASM sandbox (mod script) | Native Zig (game)     |
+----------------------------+-----------------------+
|             ECS Layer (Zig)                        |
|  Entities | Components | Systems | Scheduler       |
|  (flecs-like, generated by zGameLib patterns)      |
+----------------------------+-----------------------+
|             Compatibility Layer                    |
|  Scene tree emulation | Variant bridge             |
|  (for legacy Redot/Godot projects)                 |
+----------------------------+-----------------------+
|         Hybrid Hot-Reload Runtime                  |
|  Native (Zig) hot-swap | WASM loader & sandbox     |
+----------------------------+-----------------------+
|               Engine Systems (Zig)                 |
|  Rendering   | Physics   | Audio  | Nav            |
|  (Vulkan)    | (Jolt FFI)|        |                |
+----------------------------+-----------------------+
|               Core (Zig)                           |
|  Allocators | Math | ECS | IO | Job system         |
|  (No Variant, no GC, explicit memory)              |
+----------------------------------------------------+
|             Platform Abstraction (Zig)             |
|  Vulkan WSI | Window | Input | Threading | Files   |
+----------------------------------------------------+
```

### Key Differences & Why They Matter

| Concern | Current (C++ Redot) | Target (Zig Zodot) | Why |
|---|---|---|---|
| **Memory model** | Ref-counted + Variant boxing | Explicit allocators, flat arrays | Cache efficiency, predictable performance |
| **Game objects** | Node inheritance tree | ECS entities + components | No deep hierarchies, better cache locality |
| **Script runtime** | GDScript (interpreted) or C# (Mono/GC) | WASM sandbox | Deterministic perf, sandboxing, no GC |
| **Hot-reload** | Editor-only (C++ requires full recompile) | Native Zig hot-swap + WASM live reload | Iteration speed gains of 10-100x |
| **Build system** | SCons (Python) | `build.zig` | Native Zig integration, faster, comptime-aware |
| **Mod security** | Full engine API exposure | Capability-based WASM API | Multiplayer-safe, UGC-friendly |
| **Binary obfuscation** | Limited (stripped C++ still readable) | Zig comptime + strip | IP protection for shipped titles |
| **Dependency mgmt** | Vendored `thirdparty/` | Zig package manager + vendored C libs | Reproducible builds, easier updates |

## Jolt Physics in the Transition

Jolt Physics is already deeply integrated:

| Layer | Current | Transition |
|---|---|---|
| **Library** | Vendored C++ in `thirdparty/jolt_physics/` | Keep as C++ static lib, expose via Zig FFI |
| **Bridge** | `modules/jolt_physics/` (C++ godot module) | Replace with Zig wrapper using `@cImport` |
| **ECS integration** | Not applicable (node tree) | Physics components become ECS components. Jolt stepped queries drive system scheduling. |

Jolt will remain as the physics backend through Phase 1-2. A pure-Zig physics engine is not planned unless Jolt becomes a blocker.
